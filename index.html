001 / How I think

Most developers
ask "what tool?"
I ask "what's true?"

Before I write a single line, I need to understand what a correct solution actually requires. Not which library solves it. What needs to be true for it to work.

That question changes how you build everything. It's why my code tends to be modular, explainable, and fixable — because every piece was placed with a reason.

I'm in Semester 4. I've already built three ML systems. The gap between where I am and where most students are isn't about knowing more tools. It's about thinking differently about problems.

"The best ML engineers I've seen aren't the ones who know the most algorithms. They're the ones who know exactly when an algorithm is the wrong answer."

# How I approach every problem. No exceptions. def how_aditya_works(problem): # Step 1: What is this problem actually doing? # Not "what library fixes this" — what is true. root = understand_deeply(problem) # Step 2: Design before code. # Draw the logic. Find the edge cases. # Most bugs are born right here — in the skip. design = model_on_paper(root) # Step 3: Write clean, single-responsibility code. # If a function does two things, it does zero things. code = implement_modular(design) # Step 4: Iterate until it works in the real world. # "It runs" and "it works" are different sentences. return ship_then_improve(code) # Most people skip steps 1 and 2. # That's the entire gap.
